其中动态变化的文件可能有以下几处：

1. `config.json`
2. `.dynamic`文件夹
3. `bpnodes`文件夹

下面会详述它们分别是被哪些文件影响的。但在讲解之前，需要先介绍下注释字段。

## 注释字段说明

### 类声明

下面的示例是一个类的声明，它涉及了 4 个注释。

```javascript
/**
 * 描述信息
 * @title 机柜
 * @bpNode BPCabinet
 * @icon './images/my-bp-02.png'
 */
export class Cabinet extends THING.Entity {}
```

#### desc

`desc`是单词`description`的简写，表示自定义类的描述信息。它有两种配置方式。

一种是样例中的类的描述：

```javascript
/**
 * 描述信息
 */
export class Cabinet extends THING.Entity {}
```

另一种是使用`@desc`：

```javascript
/**
 * @desc 描述信息
 */
export class Cabinet extends THING.Entity {}
```

#### title

`@title`就是个普通的字段。

#### bpNode

`@bpNode`指的是自定义类所关联的蓝图节点。后文会详述。

#### icon

`@icon`是个特殊的字段，值是图标的路径，但是编辑器会将它解析转换为`Base64`的字符串。

如果有其它字段也需要转换为`Base64`，可以添加类型为`Base64`。

```javascript
/**
 * @image {Base64} './images/my-bp-02.png'
 */
```

> 目前这个字段只为蓝图节点服务，也就是仅在 bpnodes 文件夹下生效，它会将数据动态注入到`editorPreload.js`中。它的路径相对于插件根目录，例中`images`文件夹与`bpnodes`平级。

### 类成员声明

在自定义类型成员的注释中，采用下面注释，则可以帮助编辑器识别这个成员的信息，从而设置默认值：

* `@member` {类型} 成员名称
* `@default` 默认值
* `@instance` 声明为成员实例，不需要下接具体代码（一般用于声明父类成员）
* `@desc` 描述信息
* `@range` 取值范围
* `@step` 步进值
* `@data` 数据源（一般用于做下拉列表数据集合）
* `@bpInput` 声明为蓝图节点的输入（数据）
* `@bpOutput` 声明为蓝图节点的输出（数据）

举例：

```javascript
/**
 * @member {String} title
 * @default "爵士"
 * @desc 头衔
 * @data [
 *  "爵士",
 *  "国王",
 *  "卫士"
 * ]
 * @instance
 */

/**
 * @member {String} _brand
 * @desc 品牌
 * @data [
 *  "长城",
 *  "比亚迪",
 *  "吉利"
 * ]
 */
this._brand = '长城'

/**
 * @member {Number} _height
 * @desc 高度值
 * @range [1, 1000]
 * @step 1
 */
this._height = 100

/**
 * @member {Array<Boolean>} _slots
 * @desc 安装插槽状态
 * @range [0, 100]
 */
this._slots = []

/**
 * @member {Number} _speed
 * @desc 速度值
 * @range [0, 120]
 * @step 5
 * @bpInput
 */
this._speed = 1
```

上述例子有两点需要注意：

> 1. `@member`的值与实际代码中属性值要保持一致。
> 2. `@default` ，如果代码中给出了赋值语句，就不必添加 `@default` 了；如果同时存在，则以 `@default` 为准。而基类（`@instance`）中的属性由于没有代码部分，还是需要 `@default` 信息的。

### 类接口声明

类型的方法可以通过`@bpInput`来声明为蓝图节点中的可执行输入，举例：

```javascript
/**
 * Do something.
 * @param {Color} boundingBoxColor
 * @bpInput
 */
doSomething(boundingBoxColor) {
  // Show the boudning box
  this.helper.boundingBox.color = boundingBoxColor
  this.helper.boundingBox.visible = true
}
```

## config.json

`config.json`是整个插件的配置文件，它的整个内容中，只有一处是不允许被修改的，那就下面的`dev`部分。

```bash
{
  "name": "aa",
  "title": "标题",
  "version": "0.0.1",
  "description": "这是描述信息",
  "runtime": {
    "entrance": "main.js",
    "includes": ["libs/"]
  },
  "editor": {
    "entrance": "index.js"
  },
  "dev": {
    // 开发的配置信息（由编辑器自动生成）
    "custom": [
      // 用户自定义类列表，根据custom文件夹中代码注释生成
      {
        "cls": "Cabinet", // 自定义类导出的类名
        "desc": "机柜", // @desc这个注释，或者类的描述信息
        "title": "标题", // @title这个注释
        "bpNode": "BPCabinet", // @bpNode这个注释
        "properties": []
      }
    ],
    "bpNodes": [
      // 根据bpnodes文件夹中代码的类名生成
      "BPCabinet"
    ]
  }
}
```

`dev`中记录了两部分内容，`custom`是自定义类的元数据信息，`bpNodes`则是蓝图节点的类名。它们分别受上面所说的`custom`和`bpnodes`文件夹中代码注释的影响。

### custom

自定义类的元数据有多种。`desc`与`title`在上面已经讲过，其它的下面我们会一一解释。

#### cls

`cls`指的是自定义类导出的类名。通常有两种写法。

一种是在当前行导出：

```javascript
export class Cabinet extends THING.Entity {}
```

另一种是换行导出：

```javascript
class Cabinet extends THING.Entity {}
export { Cabinet }
```

> 我们推荐使用前者，因为后者你在修改类名时需要注意保持一致，否则编辑器会报错。  
> 不支持`export default`导出。

#### bpNode

`bpNode`指的是自定义类所关联的蓝图节点。

```javascript
/**
 * @bpNode BPCabinet
 */
export class Cabinet extends THING.Entity {}
```

它会同时生成两个文件：

1. `bpnodes/BPCabinet.js`。蓝图节点类。需要注意的是，如果命名不是 BP 开头，则不会生成；如果已有同名文件，不会进行覆盖。
2. `.dynamic/BPBaseCabinetNode.js`。上面的蓝图节点类继承的基类。命名规则是`BPBaseXXXNode`。

之后这个自定义类，如果配置了`@bpInput`，则会动态地修改蓝图节点基类的代码。

下面以一个示例来说明。这是一个自定义类：

```javascript
/**
 * @title 机柜
 * @desc 自定义机柜
 * @bpNode BPCabinet
 */
export class Cabinet extends THING.Entity {
  constructor(params = {}) {
    super(params)

    /**
     * @member {Image} _previewImageUrl
     * @desc 预览图
     * @bpInput
     */
    this._previewImageUrl = ''
  }

  /**
   * print log
   * @param {String} text
   * @param {Boolean} isLog
   * @bpInput
   */
  printLog(text, isLog) {
    console.log("It's 'Aa' custom object: ", this, text, isLog)
  }
}
```

它对应生成的`.dynamic/BPBaseCabinetNode.js`文件，内容如下：

```javascript
// Warning: 此文件为自动生成，不要直接修改，包括文件名与内容，否则会被覆盖
export class BPBaseCabinetNode extends THING.BLUEPRINT.BaseNode3D {
  constructor() {
    super()
  }

  static config = {
    name: 'BPCabinet', // 关联的蓝图节点类的名称
    inputs: [
      {
        name: 'object',
        type: 'any'
      },
      {
        // 这个是添加了`@bpInput`的属性
        name: '_previewImageUrl',
        type: 'Image'
      },
      // 以下3个是添加了`@bpInput`的方法
      {
        // 根据`@param {String} text`生成
        name: 'text',
        type: 'String'
      },
      {
        // 根据`@param {Boolean} isLog`生成
        name: 'isLog',
        type: 'Boolean'
      },
      {
        // 方法名称
        name: 'printLog',
        type: 'exec'
      }
    ]
  }

  onExecute(data, inputs, outputs) {
    let curExecName = this.curExecName

    if (curExecName == 'printLog') {
      // 添加了`@bpInput`的方法
      const text = inputs['text']
      const isLog = inputs['isLog']
      this.traverseObject(inputs['object'], (obj) => {
        obj.printLog(text, isLog)
      })
    }
  }
}
```

主要的说明都在注释中了。`@bpInput`可以添加到多个方法中，这里就不赘述了。

> 如果使用`@bpInput`的方法的参数在整个类中不能重复，既不能与属性冲突，也不能与其它方法的参数冲突。

### bpNodes

`config.json`中`bpNodes`字段比较简单，它只是蓝图节点的类名集合。

## .dynamic 文件夹

`.dynamic`包含了编辑器大部分自动生成的文件。上面已经提到蓝图节点的基类就存放这个目录下。

我们再回顾下这个文件夹还有什么。

* .dynamic 动态生成的文件夹
  * BPBaseCabinetNode.js 蓝图节点基类
  * editor.js 编辑器插件真正的入口文件
  * editorPreload.js 编辑器插件预先加载的文件，不管有没有编辑器入口文件它都加载
  * runtime.js 运行时真正的入口文件
  * runtimePreload.js 运行时预先加载

> 由于这个文件夹下的内容都是动态生成的，每次插件启动后，会进行一次编译，所以，不允许修改文件内容。

### runtimePreload.js

`runtimePreload.js`是编辑器扫描`custom`和`bpnodes`这两个文件夹后动态生成的，会在用户配置的`config.json`中的运行时入口文件之前执行。

我们看一个生成的示例：

```javascript
// Warning: 此文件为自动生成，不要直接修改，包括文件名与内容，否则会被覆盖
import { BPCabinet } from '../bpnodes/BPCabinet'
import { Cabinet } from '../custom/Cabinet'

export default function init() {
  const $registerBpNodes = (...classes) => {
    classes.forEach((cls) => {
      const parentCls = Object.getPrototypeOf(cls)
      if (parentCls.config) {
        cls.config = THING.BLUEPRINT.Utils.mergeNodeConfig(
          cls.config,
          parentCls.config
        )
      }
      THING.BLUEPRINT.Utils.registerNode(cls)
    })
  }

  $registerBpNodes(BPCabinet)

  // register customClass
  THING.Utils.registerClass('Cabinet', Cabinet, THING)
}
```

内容主要是注册蓝图节点与自定义类。

### runtime.js

`runtime.js`是真正的运行时入口文件。它的代码非常简单，引入了上面的文件和导出`config.json`中配置的`runtime/entrance`：

```javascript
// Warning: 此文件为自动生成，不要直接修改，包括文件名与内容，否则会被覆盖
import $init from './runtimePreload.js'

$init()
export * from '../main.js'
```

> 对应到`.dist`目录中，它会编译为`.dist/main.xxx.js`。

### editorPreload.js

`editorPreload.js`是编辑器扫描蓝图节点（bpnodes 文件夹）后，动态生成的文件：

```javascript
// Warning: 此文件为自动生成，不要直接修改，包括文件名与内容，否则会被覆盖
import { BPCabinet } from '../bpnodes/BPCabinet'

const $mergeBpNodeConfig = (cls) => {
  const parentCls = Object.getPrototypeOf(cls)
  if (parentCls.config) {
    cls.config = THING.BLUEPRINT.Utils.mergeNodeConfig(
      cls.config,
      parentCls.config
    )
  }
}

$mergeBpNodeConfig(BPCabinet)
editor.blueprint.registerNode(BPCabinet, {
  desc: '蓝图节点：机柜',
  path: 'earth'
})
```

> 对应到`.dist`目录中，它会被单独编译为`.dist/preload.xxx.js`。

它的作用是将蓝图节点与编辑器关联起来。

```javascript
/**
 * 蓝图节点：机柜
 * @path earth
 */
export class BPCabinet extends BPBaseCabinetNode {}
```

> 也就是说，如果你想在编辑器里使用蓝图节点信息，必须给蓝图节点类添加`@path`注释，才会产生关联。

`registerNode`的第二个参数，是蓝图节点类上的注释信息。如果还有其它注释，比如`@title`之类，编辑器未作过滤，会一并传递。这时，也可以使用上面注释说明的`@icon`。以下是个示例：

```javascript
/**
 * 蓝图节点：机柜
 * @path earth
 * @title 这是一个类
 * @icon './images/my-bp-02.png'
 * @headerColor '#ff0000'
 * @bodyColor 'rgba(100, 200, 0, 0.9)'
 */
export class BPCabinet extends BPBaseCabinetNode {}
```

这时生成的`editorPreload.js`就包含以下内容：

```javascript
editor.blueprint.registerNode(BPCabinet, {
  desc: '蓝图节点：机柜',
  path: 'earth',
  title: '这是一个类',
  icon: $icon_0,
  headerColor: '#ff0000',
  bodyColor: 'rgba(100, 200, 0, 0.9)'
})
```

其中，`$icon_0`就是转换的 base64 字符串，由于太长，这里就复制了。

### editor.js

`editor.js`与`runtime.js`相对应，它是编辑器真正的入口文件，它的内容也相对固定，引用的是`config.json`中配置的`editor/entrance`：

```javascript
// Warning: 此文件为自动生成，不要直接修改，包括文件名与内容，否则会被覆盖
import $editorDefault from '../index.js'

export default $editorDefault
```

其实从目前来看，`editor.js`并没有什么特殊的功能，直接使用`index.js`作为入口文件也没有问题，但为了后续的扩展，是有必要的。

> 对应到`.dist`目录中，它会编译为`.dist/index.xxx.js`。

它和`editorPreload.js`的关系很特别，它们都是编辑器所需要的文件，`editorPreload.js`会在`editor.js`之前执行。而编辑器的 Web 端并不是直接引入它们，而是组装了它们的`.dist/install.js`：

```javascript
import './preload.w40geAFS.js?t=1703553128217'
import './main.O_9GD7JY.js?t=1703553128217'
import plugin from './index.5v6or6JP.js?t=1703553128217'
const editor = window.editor
editor._pluginInstances.set('test002', plugin)
editor._pluginInstallTasks.push({
  onInstall: () => {
    return plugin.onInstall && plugin.onInstall()
  },
  onBeforeInstall: () => {
    return plugin.onBeforeInstall && plugin.onBeforeInstall()
  }
})
```

而在没有定义编辑器入口文件的特殊插件（比如`.common`文件夹）里，`install.js`只会引入`preload.js`和`main.js`：

```javascript
import './preload.NEbqr5gL.js?t=1703556533173'
import './main.OWmzho2X.js?t=1703556533173'
```

## bpnodes 文件夹

前文说过，如果修改了`custom`目录下的自定义类的`@bpNode`注释，如果是以`BP`开头，会在`bpnodes`文件夹下自动生成同名的蓝图节点文件，如果已有同名文件存在，不会进行覆盖处理。

```javascript
/**
 * @title 机柜
 * @desc 自定义机柜
 * @bpNode BPCabinet
 */
export class Cabinet extends THING.Entity {}
```
